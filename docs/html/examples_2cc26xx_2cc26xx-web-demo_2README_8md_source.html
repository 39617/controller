1 CC26xx Web Demo Readme
2 ======================
3 This demo project combines a number of web-based applications aiming to
4 demonstrate the CC26xx capability. The applications are:
5 
6 * A network-based UART
7 * A client for [6lbr](http://cetic.github.io/6lbr/)
8 * A CoAP server
9 * An MQTT client
10 * A web server which can be used to display sensor readings but also to
11  configure MQTT functionality
12 
13 The example has been configured to run for all CC26xx-based boards: i) The
14 SensorTag 2.0 and ii) The Srf06EB with a CC26xx or CC13xx EM mounted on it.
15 
16 To change between target boards, follow the instructions in the platform's
17 REDME file. Do not forget to `make clean` when switching between the boards.
18 
19 You can disable some of those individual components by changing the respective
20 defines in `project-conf.h`. For instance, to disable the CoAP functionality,
21 set `#define CC26XX_WEB_DEMO_CONF_COAP_SERVER 0`. The web server cannot be
22 disabled, all other aforementioned applications can.
23 
24 Network UART (net-uart)
25 -----------------------
26 This example only makes sense if you are using the Srf or if you have taken
27 the sensortag out of its case and you have it connected over JTAG to the Srf.
28 
29 The net-uart does two things:
30 
31 * When you type a string to the UART console, the string will be sent verbatim
32  to a remote UDP port 7777 listener. This can be for example a netcat listener
33  on a linux or OS X PC:
34  `nc -6ulkw 1 7777`
35 * The net-uart also listens to UDP port 7777 and when it receives a string over
36  UDP, it will print it verbatim over UART.
37 
38 The example will initially send packets to a hard-coded IPv6 address. This can
39 be changed very easily by entering a different IPv6 address to the console.
40 Thus, when the serial input string is an IPv6 address, it will not be sent as
41 any other string would, but instead it will configure the example to send to a
42 different remote address. This new IPv6 address is not persistent across
43 device reboots.
44 
45 6lbr Client
46 -----------
47 This will periodically send a UDP packet to your 6lbr, containing network
48 information, which will be used by 6lbr to construct a network graph. To see
49 this in action, fire up a browser and navigate to the 6lbr web page. The
50 default address is http://[bbbb::100]. Once the page loads, click the 'sensors'
51 tab, as per the image below.

55 CoAP Server
56 -----------
57 For this functionality to work, you will need to install the
58 [Copper (Cu)](https://addons.mozilla.org/en-US/firefox/addon/copper-270430/)
59 addon to your browser.
60 
61 From the sensors tab in the 6lbr web page, click the 'coap' link in the line
62 corresponding to your CC26xx device. Once the addon fires up, select
63 ".well-known/core" in the left pane and then hit the 'Get' button at the top.
64 
65 ![CoAP Resources](img/coap-resources.png)
66 
67 The Device will respond with a list of all available CoAP resources. This list
68 will be different between the Srf and the SensorTag. The screenshot below shows
69 a (partial) list of resources exported by the SensorTag CoAP server. Select
70 a resource on the left pane and hit 'Get' to retrieve its value. Select
71 `lt/g` and hit 'Post' to toggle the green LED, `lt/r` for the red one.
72 
73 You can also use CoAP to enable/disable BLE advertisements! Select
74 `dev/ble_advd` and then hit the "Outgoing" button in the payload panel. Type in
75 the desired payload, which can be:
76 
77 * `mode=on|off`
78 * `name=<name>`
79 * `interval=<secs>`
80 
81 or a combination of both delimited with an amp. For example, you can set as
82 payload `mode=on&name=My CC26xx Device 4&interval=5`. Once you have set the
83 payload, hit either the POST or PUT button.
84 
85 Bear in mind that you must set `name` at least once before enabling BLE
86 advertisements. If you fail to do so, the RF will refuse to enter BLE mode and
87 the CoAP engine will return 4.03 forbidden. The values of `name` and `interval`
88 persist across BLE on/off cycles, so you only have to set them once. The values
89 do _not_ persist through device powercycles.
90 
91 HTTPD
92 -----
93 Back on the 6lbr page, hit the 'web' link corresponding to your device. This
94 will take you to a web page served by the CC26xx. The HTTPD serves two pages:
95 
96 * index.html: Provides sensor readings and network information
97 * config.html: Can be used to configure the MQTT client (more below)
98 
99 In the navigation bar at the top there is also a third link, which will take
100 you directly to your device's page on IBM's quickstart service.

102 IBM Quickstart / MQTT Client
103 ----------------------------
104 The MQTT client can be used to:
105 
106 * Publish sensor readings to an MQTT broker.
107 * Subscribe to a topic and as a result receive commands from an MQTT broker
108 
109 The device will try to connect to IBM's quickstart over NAT64, so you will
110 need a NAT64 gateway in your network to make this work. A guide on how to
111 setup NAT64 is out of scope here. If this is not an option for you, you can
112 configure the device to publish to a local MQTT broker over end-to-end IPv6.
113 See below on how to change the destination broker's address.
114 
115 By default the device will publish readings to IBM's quickstart service. The
116 publish messages include sensor readings but also some other information such
117 as device uptime in seconds and a message sequence number. Click the "IBM
118 Quickstart" link in the web page to go directly to your device's page
119 on Quickstart. After a few seconds, you will see something like this:
120 
121 ![A SensorTag on IBM Quickstart](img/quickstart-sensortag.png)
122 
123 Sensor readings are only published if they have changed since the previous
124 reading (BatMon is an exception and always gets published). Additionally, you
125 can turn on/off individual readings from the config.html web page, as per the
126 figure below.
127 
128 ![Sensor Readings Configuration](img/sensor-readings-config.png)
129 
130 Some of the MQTT client functionality can be configured even further:
131 
132 * You can change the broker IP and port. This is useful if you want to use your
133  own MQTT broker instead of IBM's quickstart. The example has been tested
134  successfully with [mosquitto](http://mosquitto.org/)
135 * You can change the publish interval. Recommended values are 10secs or higher.
136  You will not be allowed to set this to anything less than 5 seconds.
137 * If you want to use IBM's cloud service with a registered device, change
138  'Org ID' and provide an 'Auth Token', which acts as a 'password', but bear in
139  mind that it gets transported in clear text, both over the web configuration
140  page as well as inside MQTT messages.
141 * The remaining configuration options are related to the content of MQTT
142  messages and in general you won't have to modify them.
143 
144 For the SensorTag, changes to the MQTT configuration get saved in external
145 flash and persist across device restarts. The same does not hold true for
146 Srf+EM builds.
147 
148 You can also subscribe to topics and receive commands, but this will only
149 work if you use "Org ID" != 'quickstart'. Thus, if you provide a different
150 Org ID (do not forget the auth token!), the device will subscribe to:
151 
152 `iot-2/cmd/+/fmt/json`
153 
154 You can then use this to toggle LEDs or to turn the buzzer on and off.
155 The buzzer is only available on the SensorTag. To do this, you can for example
156 use mosquitto client to publish to `iot-2/cmd/leds/fmt/json`. So, to turn
157 the buzzer on, you would do this:
158 
159 `mosquitto_pub -h <broker IP> -m "1" -t iot-2/cmd/buzz/fmt/json`
160 
161 Where `broker IP` should be replaced with the IP address of your mosquitto
162 broker (the one where you device has subscribed). Replace `-m "1'` with `-m "0"`
163 to turn the buzzer back off. Replace `buzz` with `leds` in the topic to change
164 the state of the LED.
165 
166 Bear in mind that, even though the topic suggests that messages are of json
167 format, they are in fact not. This was done in order to avoid linking a json
168 parser into the firmware.
