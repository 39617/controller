1 Contiki network I/O on Microsoft Windows (including the Cygwin environment) is
2 implemented based on the quite popular WinPcap library that is available at
3 [http://winpcap.org](http://winpcap.org).
4 
5 Developing Contiki network applications most likely involves working with a
6 network protocol analyzer. Wireshark (formerly known as Ethereal) is a very
7 popular one that on Windows uses - and actually comes with - the WinPcap
8 libary. Wireshark is available at [http://wireshark.org](http://wireshark.org).
9 
10 So with Wireshark installed Contiki network I/O doesn't need any additional
11 components.
12 
13 On Windows every Contiki application has one obligatory comand line argument
14 that identifies the Windows network interface to be used by Contiki. While on
15 Unix those network interfaces are called i.e. '/dev/tap0' they have on Windows
16 names like
17 
18  \Device\NPF_{F76B480A-1D31-4B3D-8002-C0EF49185737}
19 
20 In order to avoid the necessity to enter such names on the command line instead
21 the IPv4 address used by Windows is entered to identify the network interface
22 to be used by Contiki. Please note that this IPv4 address is _NOT_ the IPv4
23 address to be used by Contiki !
24 
25 Contiki network I/O on Windows uses the same MAC address used by Windows. This
26 approach often described as IP-Aliasing was primarily chosen because it avoids
27 putting the network interface into promiscuous mode. The major benefit of this
28 is the compatibility with WLAN interfaces - which mostly come with Windows
29 device drivers incapable of promiscuous mode.
30 
31 The WinPcap library works fine with the 'Microsoft Loopback Adapter' so it's
32 easy to have a Contiki network application running on Windows communicate with
33 the local Windows instance for testing purposes - and monitor the communication
34 with Wireshark.
35 
36 Windows Vista however tries to identify networks by the MAC address of the
37 default router. If that fails the network is defined as an 'Unidentified
38 Network' and thus classified as 'Public Network' resulting in very strict
39 firewall settings. As there's no default router for a loopback interface the
40 interface is always considered as a public network - which is kind of the
41 opposite of the actual situation ;-)
42 
43 Instead of fiddling with the firewall settings for 'Public Networks' (or even
44 turning the firewall completely off) there's a clean solution which defines the
45 loopback interface as not a true network interface that connects to a network.
46 This results in generally deactivating both the network identification process
47 and the firewall for the loopback interface. The details are available in the
48 Microsoft TechNet Forums thread 'Vista Network Identification for Loopback
49 Adpater' that is currently available at
50 [link](http://social.technet.microsoft.com/Forums/windows/en-US/66b42761-1b8e-4302-9134-0bb685139f4e/vista-network-identification-for-loopback-adpater)
