1 Code Contributions
2 ==================
3 
4 Do you have a new cool feature that you'd like to contribute to
5 Contiki? Or a fix for a bug? Great! The Contiki project loves code
6 contributions, improvements, and bugfixes, but we require that they
7 follow a set of guidelines and that they are contributed in a specific
8 way.
9 
10 Additional rules apply for contributions of a new hardware platform.
11 
12 General Advice
13 --------------
14 
15 The chance of getting your pull request accepted increases considerably
16 if you adhere to the following rules in addition to the aforementioned
17 formatting and naming standards:
18 
19 * Ensure that all contributed files have a valid copyright statement
20  and an open-source license.
21 * Do not bundle commits that are unrelated to each other -- create
22  separate pull requests instead.
23 * Adhere to ISO C99 in all C language source files. Exceptions are
24  allowed for those platform-dependent source files that rely on the
25  extensions of a specific set of compilers.
26 * Clean up the commit history. "git rebase -i" is useful for this purpose.
27 * Do not include executable binary files, because they are usually
28  rejected for security reasons. Instead, provide instructions for how
29  to compile the file, so that a trusted member of the merge team can
30  commit it.
31 * Write a descriptive pull request message. Explain the advantages and
32  disadvantages of your proposed changes.
33 * Before starting to work on a major contribution, discuss your idea
34  with experienced Contiki programmers (e.g., on the contiki-developers
35  mailing list) to avoid wasting time on things that have no chance of
36  getting merged into Contiki.
37 
38 Source code that goes into the mainline Contiki repository must be of
39 interest to a large part of the Contiki community. It must be
40 well-tested and the merge team must have confidence that the code can
41 be maintained over a longer period. See below for more details
42 pertaining to platform contributions.
43 
44 Contributions that have been made in research projects, and typically
45 do not get maintained thereafter, are better suited for inclusion in
46 the Contiki projects repository.
47 
48 Structuring Commits
49 -------------------
50 
51 * Write descriptive commit messages. They don't have to be very long,
52  but you should mention what the commit achieves. Commit messages
53  like "modified foo/bar.c" are not helpful, should not be used, and
54  are likely to result in you having to re-write them.
55 * Please do not add / remove irrelevant new line markers. Don't remove
56  the new line marker at the EOF.
57 * Please, make sure that your patch doesn't add lines with trailing
58  whitespaces. If you run uncrustify as discussed above, this should
59  get taken care of for you automatically.
60 * More generally speaking, make sure that each commit in your history
61  only includes changes necessary to implement whatever it is the
62  commit is trying to achieve. All changes should be mentioned in the
63  commit message.

65 Code Formatting
66 ---------------
67 
68 We require that all code contributed to the Contiki tree follows the
69 same code formatting as the existing Contiki code. We are very strict
70 on this.
71 
72 Code must be formatted according to
73 [contiki/doc/code-style.c](https://github.com/contiki-os/contiki/blob/master/doc/code-style.c).
74 
75 The Contiki source tree contains scripts to assist with correct code formatting
76 and we recommend [Uncrustify](http://uncrustify.sourceforge.net/) as the
77 preferred auto formatter. Everything is under
78 [tools/code-style](https://github.com/contiki-os/contiki/tree/master/tools/code-style).
79 
80 If you wish, you can format all changed resources in your working tree
81 automatically if the
82 [tools/code-style/uncrustify-changed.sh](https://github.com/contiki-os/contiki/blob/master/tools/code-style/uncrustify-changed.sh)
83 script is added as a [Git pre-commit
84 hook](http://git-scm.com/book/en/Customizing-Git-Git-Hooks) to your Git
85 configuration.
86 
87 Here are some examples of what you can do:
88 * To check a file's style without changing the file on disk, you can run this:
89 `./tools/code-style/uncrustify-check-style.sh <path-to-file>`
90 This script will only accept a single file as its argument.
91 
92 * To auto format a file (and change it on disk) you can run this:
93 `./tools/code-style/uncrustify-fix-style.sh <path-to-file>`
94 
95 * `uncrustify-fix-style.sh` will accept a space-delimited list of files as its argument. Thus, you can auto-format an entire directory by running something like this:
96 ``./tools/code-style/uncrustify-fix-style.sh `find cpu/cc2538 -type f -name "*.[ch]"` ``
97 
98 This is _not_ a silver bullet and developer intervention is still required. Below are some examples of code which will get misformatted by uncrustify:
99 * Math symbol following a cast to a typedef
100 ```
101  a = (uint8_t) ~P0_1; /* Cast to a typedef. Space gets added here (incorrect) */
102  a = (int)~P0_1; /* Cast to a known type. Space gets removed (correct) */
103  a = (uint8_t)P0_1; /* Variable directly after the cast. Space gets removed (correct) */
104 ```
105 
106 * `while(<condition>);` will become `while(<condition>) ;` (space incorrectly added after closing paren)
107 
108 * ﻿﻿`asm("wfi");` becomes `asm ("wfi");`: A space gets added before the opening paren, because the `asm` keyword stops this from getting interpreted as a normal function call / macro invocation. This is only a problem with `asm`. For instance, ﻿﻿`foo("bar");` gets formatted correctly.

110 Naming
111 ------
112 
113 We require that all code contributed to the Contiki tree follow the
114 Contiki source code naming standard:
115 
116 * File names are composed of lower-case characters and dashes. Like
117  this: simple-udp.c
118 * Variable and function names are composed of lower-case characters
119  and underscores. Like this: simple_udp_send();
120 * Variable and function names that are visible outside of their module
121  must begin with the name of the module. Like this:
122  simple_udp_send(), which is in the simple-udp module, declared in
123  simple-udp.h, and implemented in simple-udp.c.
124 * C macros are composed of upper-case characters and underscores. Like
125  this: PROCESS_THREAD().
126 * Configuration definitions begin with the module name and CONF_. Like
127  this: PROCESS_CONF_NUMEVENTS.

129 How to Contribute Code
130 ----------------------
131 
132 When your code is formatted according to the Contiki code style and
133 follows the Contiki naming standard, it is time to send it to the
134 Contiki maintainers to look at!
135 
136 All code contributions to Contiki are submitted as [Github pull
137 requests](https://help.github.com/articles/using-pull-requests). Pull
138 requests will be reviewed and accepted according to the guidelines
139 found in the next section.
140 
141 The basic guidelines to to start a Pull-Request:
142 * Create a new branch for your modifications. This branch should be based on the latest contiki master branch.
143 * If you already added the commits to another branch you can [cherry-pick](http://git-scm.com/docs/git-cherry-pick) them onto your new branch.
144 * Push the new branch to github.
145 * Raise the new Pull Requests on this new branch. Raising a Pull Request for the master branch is almost always a bad idea.
146 * If changes are requested do not close the pull request but rewrite your history. [Details about rewriting your history](http://git-scm.com/book/en/Git-Tools-Rewriting-History)
147 * You now force-push the changes to github. The pull-request is automatically updated.
148 
149 In Git terminology this is equivalent to:
150 * Make sure you have the original contiki repo as origin.
151 ```bash
152 $ git remote -v
153 contiki-orig https://github.com/contiki-os/contiki.git
154 ```
155 * If not add it
156 ```bash
157 $ git remote add contiki-orig https://github.com/contiki-os/contiki.git
158 ```
159 * Make sure you have the latest version of your remotes
160 ```bash
161 $ git remote update
162 ```
163 * Create a new branch "my_new_feature" based on the latest contiki master branch
164 ```bash
165 $ git checkout contiki-orig/master -b my_new_feature
166 ```
167 * Add your work. For example by cherry-picking your changes from another branch.
168 ```bash
169 $ git cherry-pick <HASH OF COMMIT>
170 ```
171 * Push to _your_ github repository
172 ```bash
173 $ git push origin my_new_feature
174 ```
175 * Make a Pull Request for that branch
176 * Rewrite your history if requested
177 ```bash
178 $ git rebase -i contiki-orig/master
179 ```
180 * As rewriting your history can break things you must force-push the changes. **Warning**: Force-pushing normally is dangerous and you might break things. Make sure you are never force-pushing branches other people are supposed to work with.
181 ```bash
182 $ git push origin my_new_feature -f
183 ```
184 * NOTE: To avoid all the pain of selectively picking commits, rebasing and force-pushing - begin your development with a branch OTHER THAN your master branch, and push changes to that branch after any local commits.

186 Pull Request Merging Policy
187 ---------------------------
188 
189 Pull requests (PRs) are reviewed by the [merge team](https://github.com/orgs/contiki-os/people).
190 Generally, PRs require two "+1" before they can be merged by someone on the merge team.
191 The since Contiki 3.0, the merging policy is the following:
192 * The PR receives **one "-1"** from a merge team member (along with specific feedback). The PR is closed. A "-1" must be accompanied with a clear explanation why the PR will not be considered for inclusion.
193 * The PR receives **two "+1"** from merge team members. The PR is merged.
194 * The PR was inactive for **two months**. A team member may either:
195  * Comment "Is there any interest for this PR? Is there any work pending on it? If not I will close it in **one month**." Back to initial state in case of activity, close otherwise.
196  * Comment "I approve this PR. If nobody disapproves within **one month**, I will merge it." Back to initial state in case of activity, merge otherwise.
197 
198 There is an exception to the rule.
199 Code that requires esoteric expertise such as some applications, platforms or tools can be merged after a single "+1" from its domain expert.

201 Travis / Regression testing
202 ---------------------------
203 
204 [Travis](https://travis-ci.org/) is a service that runs regression
205 tests. If you make a pull-request for Contiki this is automatically
206 forwarded to Travis and regression tests are run. A box with
207 information about the state of you pull request should show up after a
208 minute or two.
209 
210 If the test fails it is likely that something is wrong with your
211 code. Please look carefully at the log. It might also be that some
212 package on the testing VM was updated and causes the build to fail. If
213 you are sure that is is not your code causing the tests to fail start
214 a new issue describing the problem. Also note this in your pull
215 request.
216 
217 You can also register at [Travis](https://travis-ci.org/) for
218 free. Once you activated your Contiki repository, every push will be
219 tested at Travis. The configuration is part of the contiki repository
220 and testing will therefore work out-of-the-box. At Travis you then get
221 an overview of the state of each of your branches.

223 New Platforms
224 -------------
225 A new hardware port will be considered for inclusion in mainline Contiki
226 if it satisfies the following rules:
227 
228 * There must be at least one person willing and committed to maintain it.
229 They may but do not have to be the people who wrote the code. Similarly,
230 they may but do not have to be affiliated with the hardware manufacturer.
231 In the first instance, code maintenance would mean keeping the port up to
232 speed by submitting pull requests as Contiki moves forward. In the longer
233 term, people who maintain a reasonable level of commitment and who demonstrate
234 that they know what they're doing may be invited to become repo collaborators.
235 * The hardware must be commercially available and of interest to a wide audience.
236 In other words, ports for bespoke hardware built for e.g. a specific project /
237 a single customer / niche markets are more suitable for a Contiki fork.
238 * The code must strictly adhere to the Contiki code style, as discussed above.
239 * The new files must have a clear copyright notice and license header. Contiki's
240 preferred software license is the
241 [3-clause BSD](http://opensource.org/licenses/BSD-3-Clause).
242 Other licenses may also be considered
243 as long as they are compatible with the 3-clause BSD (e.g. the Apache 2.0 license).
244 Conversely, code distributed under GPL cannot be considered. The same applies to
245 bespoke licenses, such as those allowing use or redistribution only together with
246 certain kinds of hardware.
247 * The port must demonstrate a certain degree of completeness and maturity. Common sense
248 applies here.
249 * The port must be accompanied by examples demonstrating basic functionality. This could
250 be a set of examples under `examples/<new-hardware-port>` and/or documentation of
251 which existing examples are meant to work.
252 * The port must provide compile regression tests by extending the existing travis
253 integration testing framework. Again, we can't specify explicitly
254 what those tests should be, but something more interesting than hello-world is expected.
255 * The work must be documented. The documentation could be README.md files
256 under the platform / cpu / example dirs or wiki pages. Doxygen comments are
257 also encouraged. The documentation should include:
258  * A getting started guide, including a list of tools required to use the platform
259 (e.g. toolchain, software to program the device), where to get them from and brief notes
260 how to install them (can simply be a list of links to external guides)
261  * A list of things which will work off the shelf
262  * A list of things which are not meant to work, if any
263  * Additional reading resources (e.g. datasheets, hardware user guides, web resources)
264  * A ToDo list, if applicable.
265 * It must be possible to use the port using free software. We do not discourage the
266 use of commercial software (e.g. support for a commercial toolchain), quite the opposite.
267 However, we will insist on the existence of a free alternative for everything.
268 
269 After the port has been accepted, things meant to work off the shelf should
270 keep working off the shelf as Contiki moves forward.
271 
272 We appreciate that, for many people, contributing to Contiki is a spare time
273 activity and our expectations from port maintainers take this into
274 consideration. All we ask from maintainers is to comment on and address
275 relevant pull requests at a reasonable frequency and to make sure travis keeps
276 passing. In other words, we just want platforms to stay healthy over time and
277 to thus avoid becoming very broken / obsolete.
