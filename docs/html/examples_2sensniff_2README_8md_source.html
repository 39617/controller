1 sensniff Contiki Project
2 ========================
3 This example can be used to create an IEEE 802.15.4 wireless sniffer firmware,
4 meant to be used in parallel with
5 [sensniff](https://github.com/g-oikonomou/sensniff).
6 
7 Running
8 =======
9 * Build this example and program your device
10 * Connect your device to a host
11 * Run sensniff on your host
12 * Fire up wireshark and enjoy.
13 
14 You can run sensniff manually, or you can simply run `make sniff` from within
15 this directory. If you choose the latter option, you may have to specify the
16 port where you device is connected by using the PORT variable. For example, if
17 your device is connected to `/dev/ttyUSB1` then you should run
18 `make PORT=/dev/ttyUSB1 sniff`.
19 
20 Make sure your device's UART baud rate matches the `-b` argument passed to
21 sensniff. I strongly recommend using at least 460800. This comment does not
22 apply if your device is using native USB.
23 
24 Subsequently, make absolutely certain that your device is tuned to the same RF
25 channel as the network you are trying to sniff. You can change sniffing channel
26 through sensniff's text interface.
27 
28 More details in sensniff's README.
29 
30 Adding support for more platforms
31 =================================
32 Firstly, this example will try to turn off frame filtering and automatic h/w
33 ACKs by calling `NETSTACK_RADIO.set_value(RADIO_PARAM_RX_MODE, 0)`. If your
34 radio does not support this, then implementing this is your first step towards
35 running this example on your board.
36 
37 Secondly, in order to be able to switch channels and retrieve current/min/max
38 RF channel supported from sensniff's text interface, your device's radio driver
39 must also support:
40 
41  NETSTACK_RADIO.get_value(RADIO_PARAM_CHANNEL, ...)
42  NETSTACK_RADIO.set_value(RADIO_PARAM_CHANNEL, ...)
43  NETSTACK_RADIO.get_value(RADIO_CONST_CHANNEL_MIN, ...)
44  NETSTACK_RADIO.get_value(RADIO_CONST_CHANNEL_MAX, ...)
45 
46 The following radios have been tested:
47 
48 * CC13xx/CC26xx in PROP and IEEE modes
49 * CC2538
50 * CC2530/CC2531
51 * CC1200
52 
53 Once you have the radio sorted out, you also need to configure character I/O.
54 The firmware captures wireless frames and streams them over a serial line to
55 the host where your device is connected. This can be achieved over UART or over
56 CDC-ACM. The example makes zero assumptions about your hardware's capability,
57 you have to configure thnigs explicitly.
58 
59 * Firstly, create a directory named the same as your platform. Crate a header
60 file therein called `target-conf.h`. This will get included automatically.
61 * Then look at the header files under `pool`, perhaps your device's CPU is
62 already supported. If that's the case, then within your `target-conf.h` you
63 simply need to add a line like this:
64 
65  #define SENSNIFF_IO_DRIVER_H "pool/cc2538-io.h"
66 
67 choosing the header that corresponds to your device's CPU. Just look for any of
68 the platforms already supported to see how you can configure things in a more
69 fine-grained fashion (e.g. to select UART instance, switch between UART/USB
70 etc).
71 
72 * If your CPU is not already supported, then you need to create an additional
73 header file. In that header file, you will need to define the following three:
74 
75  #define sensniff_io_byte_out() <driver function that prints bytes>
76  #define sensniff_io_flush() <for buffered I/O. Can be empty>
77  #define sensniff_io_set_input() <driver function that sets an input callback>
78 
79 Those should map to functions implemented by your device's peripheral driver,
80 e.g. your UART driver. `_byte_out()` and `set_input()` are required, but
81 `_flush()` is optional and is only really helpful in case of drivers/hardware
82 that support buffered I/O (as is the case for some Contiki's USB drivers). Once
83 you have provided those defines, then simple go back to your `target-conf.h`
84 and:
85 
86  #define SENSNIFF_IO_DRIVER_H "header-with-my-own-defines.h"
87 
88 * The build system will also try to include `platform/Makefile.platform`. You
89 can create this Makefile if you want to extend the build system e.g. by adding
90 source files to the build, or by specifying Make variables. A common reason why
91 you may wish to do so would be to specify your device's baudrate. In doing so,
92 `make sniff` will pass the correct value as the argument to `-b`. You do not
93 have to create this file if you don't need to do so.
94 
95 That should be it!
