1 Intel Galileo Board
2 ===================
3 
4 This README file contains general information about the Intel Galileo board
5 support. In the following lines you will find information about supported
6 features as well as instructions on how to build, run and debug applications
7 for this platform. The instructions were only test in Linux environment.
8 
9 Requirements
10 ------------
11 
12 In order to build and debug the following packages must be installed in your
13 system:
14  * gcc
15  * gdb
16  * openocd
17 
18 Moreover, in order to debug via JTAG or serial console, you will some extra
19 devices as described in [1] and [2].
20 
21 Features
22 --------
23 
24 This section presents the features currently supported (e.g. device drivers
25 and Contiki APIs) by the Galileo port.
26 
27 Device drivers:
28  * Programmable Interrupt Controller (PIC)
29  * Programmable Intergal Timer (PIT)
30  * Real-Time Clock (RTC)
31  * UART
32  * Ethernet
33  * I2C
34  * GPIO (default pinmux configuration is listed in
35  platform/galileo/drivers/galileo-pinmux.c)
36  * Intel Quark X1000 SoC message bus
37  * Isolated Memory Regions (IMRs)
38 
39 Contiki APIs:
40  * Clock module
41  * Timer, Stimer, Etimer, Ctimer, and Rtimer libraries
42 
43 Standard APIs:
44  * Stdio library (stdout and stderr only). Console output through UART 1
45  device (connected to Galileo Gen2 FTDI header)
46 
47 Optional support for protection domains is also implemented and is
48 described in cpu/x86/mm/README.md.

50 Preparation
51 -----------
52 
53 Prerequisites on all Ubuntu Linux systems include texinfo and uuid-dev.
54 Additional prerequisites on 64-bit Ubuntu Linux systems include
55 gcc-multilib and g++-multilib.
56 
57 Docker can optionally be used to prepare an Ubuntu-based, containerized build
58 environment. This has been tested with Docker installed on Microsoft Windows 10.
59 
60 If not using a containerized environment, proceed to the "Building" section
61 below.
62 
63 Using a Docker-based environment on Windows requires that the repository has
64 been checked out with Git configured to preserve UNIX-style line endings. This
65 can be accomplished by changing the 'core.autocrlf' setting prior to cloning
66 the repository [5]:
67 ```
68 git config --global core.autocrlf input
69 ```
70 Note that this is a global setting, so it may affect other Git operations.
71 
72 The drive containing the repository needs to be shared with Docker containers
73 for the following steps to work [6]. Note that this is a global setting that
74 affects other containers that may be present on the host.
75 
76 Open Microsoft PowerShell and navigate to the base directory of the repository.
77 Run the following command to create the build environment:
78 ```
79 docker build -t contiki-galileo-build platform/galileo/bsp/docker
80 ```
81 This creates a container named 'contiki-galileo-build' based on Ubuntu and
82 installs development tools in the container.
83 
84 The build commands shown below can be run within the newly-created container. To
85 obtain a shell, run the following command in PowerShell from the base directory
86 of the repository.
87 ```
88 docker run -t -i -v ${Pwd}:/contiki contiki-galileo-build
89 ```
90 This command mounts the current directory and its subdirectories at the path
91 '/contiki' within the container. Changes to those files in the container are
92 visible to the host and vice versa. However, changes to the container
93 filesystem are not automatically persisted when the container is stopped.
94 
95 The containerized build environment does not currently support building the Grub
96 bootloader nor debugging using the instructions in this document.
97 
98 See the Docker Overview for more information about working with containers [7].

100 Building
101 --------
102 
103 To build applications for this platform you should first build newlib (in
104 case it wasn't already built). To build newlib you can run the following
105 command:
106 ```
107 $ ./platform/galileo/bsp/libc/build_newlib.sh
108 ```
109 
110 Once newlib is built, you are ready to build applications. By default, the
111 following steps will use gcc as the C compiler and to invoke the linker. To
112 use LLVM clang instead, change the values for both the CC and LD variables in
113 cpu/x86/Makefile.x86_common to 'clang'.
114 
115 To build applications for the Galileo platform you should set the TARGET
116 variable to 'galileo'. For instance, building the hello-world application
117 should look like this:
118 ```
119 $ cd examples/hello-world/ && make TARGET=galileo
120 ```
121 
122 This will generate the 'hello-world.galileo' file which is a multiboot-
123 compliant [3] ELF image. This image contains debugging information and it
124 should be used in your daily development.
125 
126 You can also build a "Release" image by setting the BUILD_RELEASE variable to
127 1. This will generate a Contiki stripped-image optimized for size.
128 ```
129 $ cd examples/hello-world/ && make TARGET=galileo BUILD_RELEASE=1
130 ```
131 
132 To also generate an '<application>.galileo.efi' file which is a UEFI [4] image,
133 you can run the following command prior to building applications:
134 ```
135 $ cpu/x86/uefi/build_uefi.sh
136 ```
137 
138 To restrict DMA so that peripherals are blocked from accessing memory
139 regions that do not contain any data that needs to be DMA-accessible,
140 specify X86_CONF_RESTRICT_DMA=1 as a command-line argument to the make
141 command that is used to build the image. This will configure and lock
142 the IMRs.
143 
144 Galileo Gen. 2 is targeted by default. Specify GALILEO_GEN=1 on the build
145 command line to target first generation boards.

147 Running
148 -------
149 
150 You will need a multiboot-compliant bootloader to boot Contiki images in that
151 format. However, this is not needed for booting UEFI images.
152 
153 In the bsp directory, we provide a helper script which builds the Grub
154 bootloader with multiboot support. To build the bootloader, just run the
155 following command:
156 ```
157 $ platform/galileo/bsp/grub/build_grub.sh
158 ```
159 
160 Once Grub is built, we have three main steps to run Contiki applications:
161 prepare SDcard, connect to console, and boot image. Below follows
162 detailed instructions.
163 
164 ### Prepare SDcard
165 
166 The instructions in this section are for a native Linux development environment,
167 so equivalent operations should be substituted when using some other environment
168 (e.g. Windows Explorer can be used to perform equivalent operations when using
169 Docker on Windows as a development environment).
170 
171 Mount the sdcard in directory /mnt/sdcard.
172 
173 Create UEFI boot directory:
174 ```
175 $ mkdir -p /mnt/sdcard/efi/boot
176 ```
177 
178 #### Approach for Multiboot-compliant ELF Image
179 
180 Copy Contiki binary image to sdcard
181 ```
182 $ cp examples/hello-world/hello-world.galileo /mnt/sdcard
183 ```
184 
185 Copy grub binary to sdcard
186 ```
187 $ cp platform/galileo/bsp/grub/bin/grub.efi /mnt/sdcard/efi/boot/bootia32.efi
188 ```
189 
190 #### Approach for UEFI Image
191 
192 Copy Contiki binary image to sdcard:
193 ```
194 $ cp examples/hello-world/hello-world.galileo.efi /mnt/sdcard/efi/boot/bootia32.efi
195 ```
196 
197 ### Connect to the console output
198 
199 Connect the serial cable to your computer as shown in [8] for first generation
200 boards and [2] for second generation boards.
201 
202 Choose a terminal emulator such as PuTTY. Make sure you use the SCO keyboard
203 mode (on PuTTY that option is at Terminal -> Keyboard, on the left menu).
204 Connect to the appropriate serial port using a baud rate of 115200.
205 
206 ### Boot Contiki Image
207 
208 Turn on your board. After a few seconds you should see the following text
209 in the screen:
210 ```
211 Press [Enter] to directly boot.
212 Press [F7] to show boot menu options.
213 ```
214 
215 Waiting for the system to select the default boot device may be sufficient.
216 However, if this does not boot Contiki or Grub (depending on what is installed
217 as the UEFI boot image) then perform the following procedure after rebooting
218 and waiting for the boot message to appear: Press <F7> and select the option
219 "UEFI Misc Device" within the menu.
220 
221 No additional steps should be required to boot a Contiki UEFI image.
222 
223 Run the following additional commands to boot a multiboot-compliant image:
224 ```
225 $ multiboot /hello-world.galileo
226 $ boot
227 ```
228 
229 ### Verify that Contiki is Running
230 
231 This should boot the Contiki image, resulting in the following messages being
232 sent to the serial console:
233 ```
234 Starting Contiki
235 Hello, world
236 ```

238 Debugging
239 ---------
240 
241 This section describes how to debug Contiki via JTAG. The following
242 instructions consider you have the devices: Flyswatter2 and ARM-JTAG-20-10
243 adapter (see [1]).
244 
245 Attach the Flyswatter2 to your host computer with an USB cable. Connect the
246 Flyswatter2 and ARM-JTAG-20-10 adapter using the 20-pins head. Connect the
247 ARM-JTAG-20-10 adapter to Galileo Gen2 JTAG port using the 10-pins head.
248 
249 Once everything is connected, run Contiki as described in "Running" section,
250 but right after loading Contiki image (multiboot command), run the following
251 command:
252 ```
253 $ make TARGET=galileo debug
254 ```
255 
256 The 'debug' rule will run OpenOCD and gdb with the right parameters. OpenOCD
257 will run in background and its output will be redirected to a log file in the
258 application's path called LOG_OPENOCD. Once gdb client is detached, OpenOCD
259 is terminated.
260 
261 If you use a gdb front-end, you can define the "GDB" environment
262 variable and your gdb front-end will be used instead of default gdb.
263 For instance, if you want to use cgdb front-end, just run the command:
264 ```
265 $ make BOARD=galileo debug GDB=cgdb
266 ```
