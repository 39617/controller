1 # CC13xx/CC26xx Very Sleepy Demo
2 
3 This example demonstrates a way of deploying a very low-consuming, very sleepy
4 node. The node has two modes of operation:
5 
6 * Normal: ContikiMAC duty-cycles the radio as usual. The node is reachable.
7 * Very Sleepy: Radio cycling mostly off, except when we need to perform network
8  maintenance tasks. In this mode, the node is unreachable for most of the time.
9 
10 The node will operate in RPL leaf mode. This means that it will be reachable
11 downwards, but it will not advertise the DODAG and it will not participate in
12 routing.
13 
14 After booting, the node will enter "normal" mode.
15 
16 The node exposes an OBSERVEable CoAP resource. It will notify subscribers with
17 a new value for this resource every `interval` seconds. It will then stay in
18 normal mode for `duration` seconds. During this time window, it will be
19 reachable over the network in order to e.g. receive a new configuration.
20 When this time window expires, the node will switch back to very sleepy mode.
21 This will only happen if very sleepy mode has been enabled by setting `mode=1`
22 as per the instructions below.
23 
24 When the node is duty-cycling the radio, either because it is in normal mode or
25 because network maintenance is taking place, it will keep its green LED on thus
26 providing an indication that it is reachable.
27 
28 A normal mode stint can be manually triggered by pressing the left button.
29 
30 ## Requirements
31 
32 To run this example you will need:
33 
34 * A border router operating with the same RDC, same channel, same radio mode
35  (e.g. IEEE or sub-ghz), same PAN ID. Alternatively, you can
36  use [6lbr](https://github.com/cetic/6lbr) with a suitable slip-radio.
37 * The [Copper (Cu)](https://addons.mozilla.org/en-US/firefox/addon/copper-270430/)
38  addon for Firefox
39 
40 ## Configuration
41 
42 To configure the node, send a CoAP POST message to the `very_sleepy_config`
43 resource. The POST message's payload must specify _at least one_ of:
44 
45 * `mode=0|1`: Send `mode=1` to enable very sleepy mode, `mode=0` to disable it.
46 * `interval=n` where `n` is the number of seconds between two consecutive normal
47  mode periods. This interval also dictates the OBSERVEr notification period.
48 * `duration=n` where `n` is the number of seconds that the node will stay in
49  normal mode before dropping to very sleepy mode. This value is only relevant
50  if `mode==1`.
51 
52 A POST request must contain at least one of the above, but they are otherwise
53 all optional. So, for example, a POST may simply specify `interval=n`. To send
54 multiple values, delimit them with `&`. So you can send something like
55 `mode=1&interval=60&duration=20`
56 
57 The current running configuration can be retrieved by sending a GET request to
58 the same CoAP resource.
59 
60 ## Running the example
61 
62 * Deploy your border router or 6lbr
63 * Turn on the very sleepy node.
64 * Fire up the Copper addon
65 * Select `.well-known/core` and hit `GET`
66 * Configure very sleepy operation:
67  * Select the `very_sleepy_config` resource
68  * In the `Outgoing` pane, type your POST payload as per the instructions
69  above. For example, you can type: `mode=1&interval=30&duration=10`
70  * Hit `POST`
71 * Select the `sen/readings` resource and hit `OBSERVE`
72 
73 ## Caveats
74 
75 If you click on a resource in the Copper resources tree while you are observing
76 a different resource, the OBSERVEr for the latter will be stopped without
77 notifying the CoAP server. This will result in the server sending out OBSERVE
78 notifications that will be responded to with port unreachable ICMPv6 messages.
79 This will continue for quite a while, until the server detects that the
80 OBSERVEr has been lost (a test currently performed once every 20 notifications).
81 In order to prevent this from happening, hit the "Cancel" button for the
82 OBSERVE before switching views to a different resource. This will unregister
83 the observer.
84 
85 In very sleepy mode, the radio is not truly always off. The contiki core needs
86 to perform other periodic tasks in order to maintain network connectivity. For
87 that reason, this example will allow the radio to turn on periodically even
88 while in very sleepy mode. Thus, you may see that the node becomes briefly
89 reachable every now and then. However, do not count on those periods of
90 reachability to perform any tasks, as they will be brief and will be disrupted
91 without warning.
