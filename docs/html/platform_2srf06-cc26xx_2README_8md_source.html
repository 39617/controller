1 Getting Started with Contiki for TI CC26xx
2 ==========================================
3 
4 This guide's aim is to help you start using Contiki for TI's CC26xx. The
5 platform supports two different boards:
6 
7 * SmartRF 06 Evaluation Board with a CC26xx or CC13xx Evaluation Module
8  (relevant files and drivers are under `srf06/`)
9 * CC2650 SensorTag 2.0 (relevant drivers under `sensortag/cc2650`)
10 * CC2650 LaunchPad (relevant drivers under `launchpad/cc2650`)
11 * CC1310 LaunchPad (relevant drivers under `launchpad/cc1310`)
12 
13 The CPU code, common for both platforms, can be found under `$(CONTIKI)/cpu/cc26xx-cc13xx`.
14 The port was developed and tested with CC2650s, but the intention is for it to
15 work with the CC2630 as well. Thus, bug reports are welcome for both chips.
16 Bear in mind that the CC2630 does not have BLE capability. Similar rules apply
17 in terms of CC13xx chips.
18 
19 This port is only meant to work with 7x7mm chips
20 
21 This guide assumes that you have basic understanding of how to use the command
22 line and perform basic admin tasks on UNIX family OSs.
23 
24 Port Features
25 =============
26 The platform has the following key features:
27 
28 * Deep Sleep support with RAM retention for ultra-low energy consumption.
29 * Support for CC26xx RF in IEEE as well as BLE mode (BLE support is very basic
30  since Contiki does not provide a BLE stack).
31 * Support for CC13xx prop mode: IEEE 802.15.4g-compliant sub GHz operation
32 
33 In terms of hardware support, the following drivers have been implemented:
34 
35 * SmartRF06 EB peripherals
36  * LEDs
37  * Buttons
38  * UART connectivity over the XDS100v3 backchannel
39 * SensorTag 2.0
40  * LEDs
41  * Buttons (One of the buttons can be used as a shutdown button)
42  * Reed relay
43  * Motion Processing Unit (MPU9250 - Accelerometer, Gyro)
44  * BMP280 sensor
45  * TMP007 sensor
46  * HDC1000 sensor
47  * OPT3001 sensor
48  * Buzzer
49  * External SPI flash
50 * LaunchPads
51  * LEDs
52  * Buttons
53  * External SPI flash

55 Requirements

92 Build your First Example
93 ------------------------
94 It is recommended to start with the `cc26xx-demo` example under `examples/cc26xx/`. This is a very simple example which will help you get familiar with the hardware and the environment. This example can be used for the Sensortag and SmartRF06 EB.
95 
96 Strictly speaking, to build it you need to run `make TARGET=srf06-cc26xx BOARD=srf06/cc26xx`. However, the example directories contain a `Makefile.target` which is automatically included and specifies the correct `TARGET=` argument. The `BOARD=` environment variable defaults to `srf06/cc26xx` (which is the SmartRF06 EB + CC26XXEM). Thus, for examples under the `cc26xx` directory, and when using the SmartRF06 EB, you can simply run `make`.
97 
98 Other options for the `BOARD` make variable are:
99 
100 * Srf06+CC26xxEM: Set `BOARD=srf06/cc26xx`
101 * Srf06+CC13xxEM: Set `BOARD=srf06/cc13xx`
102 * CC2650 tag: Set `BOARD=sensortag/cc2650`
103 * CC2650 Launchpad: Set `BOARD=launchpad/cc2650`
104 * CC1310 Launchpad: Set `BOARD=launchpad/cc1310`
105 
106 If the `BOARD` variable is unspecified, an image for the Srf06 CC26XXEM will be built.
107 
108 If you want to switch between building for one platform to the other, make certain to `make clean` before building for the new one, or you will get linker
109 errors.
110 
111 For the `cc26xx-demo`, the included readme describes in detail what the example does.
112 
113 To generate an assembly listing of the compiled firmware, run `make cc26xx-demo.lst`. This may be useful for debugging or optimizing your application code. To intersperse the C source code within the assembly listing, you must instruct the compiler to include debugging information by adding `CFLAGS += -g` to the project Makefile and rebuild by running `make clean cc26xx-demo.lst`.

115 How to Program your Device
116 --------------------------
117 To program your device on Windows, use TI's [SmartRF Flash Programmer][smart-rf-flashprog] (FLASH-PROGRAMMER-2).
118 
119 On Linux and OS X, you can program your device via the chip's serial ROM boot loader. In order for this to work, the following conditions need to be met:
120 
121 * The board can support the bootloader. This is the case for SmartRF06EB with CC26xx/CC13xx EMs and it is also the case for LaunchPads. Note that uploading over serial does not (and will not) work for the Sensortag.
122 * The chip is not programmed with a valid image, or the image has the bootloader backdoor unlocked. To enable the bootloader backdoor in your image, define `ROM_BOOTLOADER_ENABLE` to 1 in `contiki-conf.h`.
123 
124 You will then need to manually enter the boot loader and use the `.upload` make target (e.g. `make cc26xx-demo.upload` for the `cc26xx-demo`). On the SmartRF06, you enter the boot loader by resetting the EM (EM RESET button) while holding the `select` button. For the LaunchPad, you enter the bootloader by resetting the chip while holding `BTN_1`. It is possible to change the pin and its corresponding level (High/Low) that will trigger bootloader mode by changing `SET_CCFG_BL_CONFIG_BL_LEVEL` and `SET_CCFG_BL_CONFIG_BL_PIN_NUMBER` in `board.h`.
125 
126 The serial uploader script will automatically pick the first available serial port. If this is not the port where your node is connected, you can force the script to use a specific port by defining the `PORT` argument eg. `make cc26xx-demo.upload PORT=/dev/tty.usbserial`.
127 
128 For more information on the serial bootloader, see its README under the `tools/cc2538-bsl` directory.

130 CC13xx/CC26xx Border Router over UART
131 =====================================
132 The platform code can be used as a border router (SLIP over UART) by using the
133 example under `examples/ipv6/rpl-border-router`. This example defines the
134 following:
135 
136  #ifndef UIP_CONF_BUFFER_SIZE
137  #define UIP_CONF_BUFFER_SIZE 140
138  #endif
139 
140  #ifndef UIP_CONF_RECEIVE_WINDOW
141  #define UIP_CONF_RECEIVE_WINDOW 60
142  #endif
143 
144 The CC26xx port has much higher capability than some other platforms used as
145 border routers. Thus, before building the example, it is recommended to delete
146 these two configuration directives. This will allow platform defaults to take
147 effect and this will improve performance and stability.
148 
149 Do not forget to set the correct channel by defining `RF_CORE_CONF_CHANNEL` as
150 required.

161  #undef UIP_CONF_BUFFER_SIZE

175 Sensortag UART usage (with or without the Debugger Devpack)
176 ===========================================================
177 There are two ways to get debugging (printf etc) output from the Sensortag.
178 
179 * Purchase a Debugger Devpack and set `BOARD_CONF_DEBUGGER_DEVPACK` to 1 in
180 `contiki-conf.h` or `project-conf.h`. This will work off the shelf for revision
181 1.2.0 of the debugger devpack.
182 * If you have an older (rev 1.0.0) devpack, you will need to do the above and
183 then to modify `board.h` in order to cross the RX and TX DIO mappings. (TX to
184 `IOID_28`, RX to `IOID_29`).
185 * If you don't have/want a debugger devpack, you can use a SmartRF and modify
186 the jumper configuration on P408 as discussed in
187 [this thread](https://e2e.ti.com/support/wireless_connectivity/f/158/p/411992/1483824#1483824)
188 on E2E. For this to work, you need to set `BOARD_CONF_DEBUGGER_DEVPACK` to 0.

190 IEEE vs Sub-GHz operation
191 =========================
192 The platform supports both modes of operation, provided the chip also has the
193 respective capability. If you specify nothing, the platform will default to
194 Sub-GHz mode for CC13xx devices, IEEE mode otherwise. To force IEEE mode, you
195 need to add this line to your `project-conf.h`.
196 
197  #define CC13XX_CONF_PROP_MODE 0

199 Low Power Operation
200 ===================
201 The platform takes advantage of the CC26xx's power saving features. In a
202 nutshell, here is how things work:
203 
204 * When the RF is TXing, the CPU will enter sleep mode and will resume after TX
205  has completed.
206 * When there are no events in the Contiki event queue, the chip will enter
207  'some' low power mode (more below).
208 
209 We do not use pre-defined power profiles (e.g. as mentioned in the TRM or as
210 we do for the CC2538 with LPM1, LPM2 etc). Each time we enter low power
211 operation, we either put the CM3 to sleep or to deep sleep. The latter case is
212 highly configurable: the LPM engine allows other code modules to register
213 themselves for notifications and to configure low power operation. With these
214 facilities, a module can e.g. prohibit deep sleep altogether, or it can request
215 that a power domain be kept powered. The LPM engine will turn off as many
216 CC26xx components as it can while satisfying all restrictions set by registered
217 modules.
218 
219 To determine which power mode to use, the following logic is followed:
220 
221 * The deepest available low power mode can be hard-coded by using
222  the `LPM_MODE_MAX_SUPPORTED` macro in the LPM driver (`lpm.[ch]`). Thus, it
223  is possible to prohibit deep sleep altogether.
224 * Code modules which are affected by low power operation can 'register'
225  themselves with the LPM driver.
226 * If the projected low-power duration is lower than `STANDBY_MIN_DURATION`,
227  the chip will simply sleep.
228 * If the projected low power duration is sufficiently long, the LPM will visit
229  all registered modules to query the maximum allowed power mode (maximum means
230  sleep vs deep sleep in this context). It will then drop to this power mode.
231  This is where a code module can forbid deep sleep if required.
232 * All registered modules will be notified when the chip is about to enter
233  deep sleep, as well as after wake-up.
234 
235 When the chip does enter deep sleep:
236 
237 * The RF Core, VIMS, SYSBUS and CPU power domains are always turned off. Due to
238  the way the RF driver works, the RFCORE PD should be off already.
239 * Peripheral clocks stop
240 * The Serial and Peripheral power domains are turned off, unless an LPM module
241  requests them to stay operational. For example, the net-uart demo keeps the
242  serial power domain powered on and the UART clocked under sleep and deep
243  sleep in order to retain UART RX functionality.
244 * If both SERIAL and PERIPH PDs are turned off, we also switch power source to
245  the uLDO for ultra low leakage under deep sleep.
246 
247 The chip will come out of low power mode by one of the following events:
248 
249 * Button press or, in the case of the SensorTag, a reed relay trigger
250 * Software clock tick (timer). The clock ticks at 128Hz, therefore the maximum
251  time we will ever spend in a sleep mode is 7.8125ms. In hardware terms, this
252  is an AON RTC Channel 2 compare interrupt.
253 * Rtimer triggers, as part of ContikiMAC's sleep/wake-up cycles. The rtimer
254  sits on the AON RTC channel 0.
